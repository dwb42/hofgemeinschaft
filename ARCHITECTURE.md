# Architecture

Stand: 2026-07-01

## Overview

This repository is an npm workspace monorepo with a static React frontend and a small Express production server.

```text
apps/
  web/      React + TypeScript + Vite frontend
  server/   Node.js + Express + TypeScript server
qa/
  screenshots/
  design-review/
```

The current site is content-led and mostly static, but the separated web/server structure keeps the project ready for later backend features.

## Frontend Architecture

The frontend lives in `apps/web`.

Key conventions:

- Routes live in `apps/web/src/routes/`.
- Reusable UI components live in `apps/web/src/components/`.
- Page sections live in `apps/web/src/sections/`.
- Page content and SEO metadata are centralized in `apps/web/src/content/pages.ts`.
- Site-wide config is centralized in `apps/web/src/config/siteConfig.ts`.
- Image imports are centralized in `apps/web/src/config/images.ts`.
- Global CSS lives in `apps/web/src/styles/global.css`.

React Router defines the four current public routes in `apps/web/src/App.tsx`.

## Styling Architecture

The project currently uses one global CSS file rather than CSS modules or a component library.

`apps/web/src/styles/global.css` contains:

- `:root` design tokens for colors, max width, radius and shadows
- global element defaults
- header and mobile drawer styles
- hero layout and image overlay styles
- section, card, checklist and contact styles
- footer styles
- responsive media queries

Current breakpoints:

- `@media (max-width: 900px)` switches navigation and major grids toward mobile/tablet behavior.
- `@media (max-width: 700px)` handles phone-specific hero, spacing and button layout.
- `@media (max-width: 390px)` tightens the smallest phone hero typography.

The design uses real HTML text over images. Important text should not be baked into bitmap assets.

## Server Architecture

The server lives in `apps/server`.

Current behavior:

- Express app setup: `apps/server/src/index.ts`
- server config: `apps/server/src/config/serverConfig.ts`
- health route: `apps/server/src/routes/health.ts`
- future database placeholder: `apps/server/src/db/`

The production server serves the built frontend from `apps/web/dist` and uses a catch-all route to return `index.html`, allowing React Router routes to load directly.

## Data and Persistence

There is no active persistence in V1.

There are no contact forms, submissions, users, authentication, admin tools or database tables. SQLite is only reserved as a future integration direction under `apps/server/src/db/`.

If future sessions add persistence, document the schema and migration strategy before adding application behavior that depends on it.

## QA Tooling

General QA documentation lives in `qa/README.md`.

Design review screenshots are generated by:

```text
qa/design-review/create-screenshots.mjs
```

The npm script is:

```bash
npm run screenshots:design-review
```

The script captures all four routes at the review viewport set and writes output to:

```text
qa/design-review/current/
```

The script assumes the site is already running. It defaults to `http://localhost:4173` and supports `DESIGN_REVIEW_BASE_URL`.

## Long-Term Patterns

- Keep contact data centralized in `siteConfig.ts`.
- Keep public page copy and SEO metadata centralized in `pages.ts`.
- Add images through `config/images.ts` rather than importing them ad hoc across routes.
- Keep V1 contact behavior form-free unless the product direction changes explicitly.
- Preserve the separate frontend/server boundary even while the site remains mostly static.
- Store reproducible QA and design review artifacts under `qa/`.
